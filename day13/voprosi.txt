Основная цель выделения сервисного слоя — отделение бизнес-логики от представления (контроллеров) и инфраструктуры (БД, внешние API), что обеспечивает переиспользование кода, упрощает тестирование, централизует управление транзакциями и делает архитектуру гибкой для изменений.

DTO (Data Transfer Object) — это простой объект-контейнер для передачи данных между слоями (например, между сервисом и контроллером), не содержащий поведения и бизнес-логики. Domain Model — это объект, содержащий бизнес-логику, правила валидации, методы для изменения состояния и отражающий сущности предметной области. Основное отличие: DTO служит для транспорта данных, Domain Model — для реализации бизнес-правил.

Unit of Work — это паттерн, который группирует несколько операций с репозиториями в одну атомарную транзакцию, обеспечивая, что все изменения либо сохранятся вместе, либо откатятся при ошибке. Он нужен для поддержания согласованности данных при сложных бизнес-сценариях, включающих обновление нескольких сущностей.

Repository — паттерн, отвечающий за доступ к данным: предоставляет интерфейс для CRUD-операций и запросов к хранилищу, скрывая детали реализации БД. Service — паттерн, содержащий бизнес-логику, координирующий работу нескольких репозиториев, применяющий правила предметной области и управляющий транзакциями. Разница: Repository работает с данными, Service — с бизнес-правилами.

Внедрение зависимостей (DI) решает проблему жёсткой связанности классов, когда сервис сам создаёт свои зависимости (например, репозитории через new), что затрудняет тестирование, замену реализаций и поддержку. DI позволяет передавать зависимости извне (через конструктор, сеттер), делая код гибким, тестируемым и легко конфигурируемым.

Бизнес-логику нельзя использовать в контроллерах, потому что это нарушает принцип единственной ответственности, делает контроллеры "толстыми" и сложными для тестирования, дублирует код, привязывает бизнес-правила к фреймворку и усложняет их переиспользование в других контекстах (например, в CLI или API). Контроллер должен только принимать запросы, вызывать сервис и возвращать ответ.

В сервисном слое должны быть кастомные бизнес-исключения (например, OrderValidationError, InsufficientFundsException, EntityNotFoundException), отражающие предметную область, а также обёртки над инфраструктурными ошибками (например, DatabaseConnectionError). Они не должны напрямую использовать низкоуровневые исключения (SQLAlchemyError, ConnectionError) — их нужно транслировать в понятные бизнесу ошибки на уровне сервиса.

Протестировать сервис без подключения к реальной БД можно с помощью моков (mock) репозиториев, где подменяются методы на заранее определённые возвращаемые значения, и подставных (in-memory) реализаций репозиториев, хранящих данные в словаре. Это позволяет проверять бизнес-логику изолированно, быстро и без внешних зависимостей. Также можно использовать тестовый контейнер с in-memory БД (SQLite) для интеграционных тестов.
