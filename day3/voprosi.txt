1. Связь M:N требует промежуточной таблицы, потому что в реляционной БД нельзя хранить список значений в одном поле. В моём варианте «Гостиница»: бронирование может включать много услуг, услуга может быть во многих бронированиях — таблица Доп_услуги_бронирования.

2. В промежуточной таблице Доп_услуги_бронирования три атрибута: id_бронирования (FK), id_услуги (FK) и количество.

3. ON DELETE RESTRICT — запрещает удаление родительской записи, если есть дочерние. ON UPDATE CASCADE — автоматически обновляет FK в дочерней таблице при изменении PK в родительской.

4. Выделить два поля (Ctrl+клик) → правой кнопкой → Set Column(s) as Primary Key.

5. Чтобы ID генерировался автоматически при добавлении новых записей, без ручного ввода.

6. File → Export → Export as PNG/SVG/PDF → выбрать PDF → сохранить.

7. 3НФ — неключевые поля зависят только от первичного ключа, нет транзитивных зависимостей. Да, моя диаграмма удовлетворяет 3НФ.
